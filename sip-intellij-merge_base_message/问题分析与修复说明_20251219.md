## ⚠️ 问题3：画面撕裂问题（新出现）

### 问题描述

用户反馈：
> "现在可以加载画面了，但是又出现了以前没出现过的部分撕裂"

### 问题根本原因

这是我修复时引入的**副作用**：

1. **原因分析**：
   - 修复后，grabber一直保持启动状态（不再释放）
   - 停止采集时，grabber内部缓冲区中残留旧帧
   - 第二次通话开始时，可能先读取到缓冲区中的旧帧
   - 导致新旧帧混合，产生撕裂效果

2. **为什么以前没出现**：
   - 以前每次通话结束后都会释放grabber
   - 第二次通话会完全重新初始化
   - 不会有旧帧残留

### 修复方案

**修改文件**: `VideoCapture.java`
**修改位置**: 第143-180行（stopCapture方法）

```java
public void stopCapture() {
    if (!isCapturing) {
        return;
    }

    log.info("停止视频采集");
    isCapturing = false;

    // 等待采集线程结束
    if (captureThread != null) {
        try {
            captureThread.join(1000);
        } catch (InterruptedException e) {
            log.warn("等待采集线程结束时被中断", e);
        }
    }

    // ⚡ 修复画面撕裂问题：清空grabber缓冲区，避免下次启动时读取旧帧
    if (grabber != null && isPrewarmed) {
        try {
            log.debug("清空grabber缓冲区...");
            // 抓取并丢弃3帧，清空缓冲区中的旧帧
            for (int i = 0; i < 3; i++) {
                Frame oldFrame = grabber.grab();
                if (oldFrame != null) {
                    // 立即释放帧资源
                    oldFrame.close();
                }
            }
            log.debug("✅ grabber缓冲区已清空");
        } catch (Exception e) {
            log.warn("⚠️ 清空grabber缓冲区失败: {}", e.getMessage());
        }
    }

    log.info("视频采集已停止（grabber保持预热状态，可重用）");
}
```

### 修复原理

1. **缓冲区清理**：停止采集后，抓取并丢弃3帧旧数据
2. **资源释放**：调用 `frame.close()` 释放旧帧内存
3. **保持预热**：grabber仍然保持启动状态，可快速重用
4. **兼顾性能和质量**：既保持快速启动，又避免画面撕裂

### 预期效果

✅ 第二次、第三次通话不再出现画面撕裂
✅ 保持快速通话建立（使用预热的grabber）
✅ 画面质量正常，无新旧帧混合
