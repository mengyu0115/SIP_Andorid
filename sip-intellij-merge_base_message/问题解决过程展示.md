# SIP项目问题解决过程展示 - PPT汇报材料

## 问题1：挂断通话后无法再次发起通话

### 问题描述
第一次通话正常挂断后，尝试进行第二次通话时：
- 主叫方（user101）拨号后收到180 Ringing响应
- 被叫方（user100）客户端**完全没有反应**，未弹出来电对话框
- 系统日志显示"当前有通话进行中，无法发起新的呼叫"
- 通话状态持续卡在RINGING，无法恢复

### 根本原因分析
通过日志追踪发现了**监听器覆盖问题**：

```
时间轴分析：
20:45:37 - 第一次通话结束，资源清理完成
20:45:37 - CallWindowController窗口安排2秒后关闭
20:45:47 - 第二次INVITE到达（10秒后）
20:45:47 - CallWindowController接收到来电事件
         ❌ 但只记录日志，不显示来电对话框
20:46:23 - 用户手动尝试拨号
         ❌ 被拒绝："当前有通话进行中"
```

**核心技术问题**：
1. `CallWindowController.initialize()` 中调用 `setupCallEventListener()`
2. 这会**替换掉** MainController 的 CallEventListener
3. CallWindowController 的 `onIncomingCall()` 方法**只记录日志**，代码注释说"在MainController中处理"
4. 第一次通话结束后，窗口关闭，但**原始监听器从未被恢复**
5. 第二次来电被CallWindowController监听器接收，但不处理，用户看不到任何提示

### 解决方案

采用**监听器保存-恢复机制**：

**1. 保存原始监听器**（CallWindowController.java）
```java
// 在类成员变量中保存原始监听器
private SipCallManager.CallEventListener originalListener;

// 初始化时保存MainController的监听器
public void initialize() {
    originalListener = callManager.getCurrentCallEventListener();
    log.info("✅ 已保存原始CallEventListener");
    setupCallEventListener(); // 设置自己的监听器
}
```

**2. 通话结束时恢复监听器**
```java
private void restoreOriginalListener() {
    if (originalListener != null) {
        log.info("✅ 恢复原始CallEventListener");
        callManager.setCallEventListener(originalListener);
    } else {
        log.warn("⚠️ 原始CallEventListener为null");
    }
}

// 在三个关键位置调用恢复方法：
// - 通话正常结束时
// - 通话失败时
// - 用户手动挂断时
```

**3. 在SipCallManager中添加getter方法**
```java
public CallEventListener getCurrentCallEventListener() {
    return this.callEventListener;
}
```

### 修复效果

✅ 第一次通话结束后，MainController的监听器被正确恢复
✅ 第二次来电到达时，用户能看到来电对话框
✅ 支持连续多次通话（2次、3次、N次）
✅ 通话状态管理正确，不会卡在RINGING

## 问题2：消息发送没有同步

### 问题描述

客户端之间发送SIP MESSAGE时出现同步问题：
- 发送方显示消息已发送，但接收方收到延迟或未收到
- 部分消息发送失败，MSS服务器返回 **401 Unauthorized** 错误
- 消息丢失率约 30%，用户体验极差
- 测试发现：有些消息立即送达，有些消息永远收不到

### 根本原因分析

**问题核心：缺少SIP摘要认证（Digest Authentication）**

```
典型失败日志：
[ERROR] ❌ [消息发送失败] 401 Proxy Authentication Required
[WARN]  ⚠️ MSS服务器要求认证，但客户端未响应
[ERROR] 消息未送达：Call-ID=abc123...
```

**技术原因深度分析**：

1. **SIP MESSAGE请求缺少认证头**
   ```java
   // SipMessageManager.java (早期版本 - 第350-360行)
   Request request = getMessageFactory().createRequest(
       requestURI,
       Request.MESSAGE,
       callIdHeader,
       cSeqHeader,
       fromHeader,
       toHeader,
       viaHeaders,
       maxForwards
   );
   // ❌ 直接发送，没有 Authorization header
   transaction.sendRequest();
   ```

2. **未处理401质询响应**
   ```java
   // 早期版本的 processResponse (第551-572行)
   public void processResponse(ResponseEvent responseEvent) {
       if (statusCode == Response.OK) {
           log.info("✅ [消息送达确认] 200 OK");
       } else if (statusCode == 401) {
           log.error("❌ [消息发送失败] 401 {}", response.getReasonPhrase());
           // ❌ 只记录错误，没有重发逻辑！
       }
   }
   ```

3. **MSS服务器的SIP认证流程**
   - 第1次请求：客户端发送MESSAGE（无认证）
   - 服务器响应：`401 Unauthorized` + `WWW-Authenticate` header
   - **期望行为**：客户端重发MESSAGE + `Authorization` header
   - **实际行为**：客户端直接放弃，消息丢失

### 解决方案

**核心策略：实现标准的SIP摘要认证挑战-响应机制**

**1. 保存原始请求用于重发**

在发送MESSAGE时保存请求到待处理队列（SipMessageManager.java 第368-370行）：
```java
// 保存请求（用于401重发）
CallIdHeader callIdHeader2 = (CallIdHeader) request.getHeader(CallIdHeader.NAME);
pendingRequests.put(callIdHeader2.getCallId(), request);

log.info("📤 正在发送SIP MESSAGE请求...");
transaction.sendRequest();
```

**2. 实现401认证挑战处理器**

在 `SipMessageManager.java` 中添加认证处理逻辑（第577-635行）：
```java
/**
 * 处理401/407认证质询
 */
private void handleAuthenticationChallenge(ResponseEvent responseEvent) {
    try {
        Response response = responseEvent.getResponse();
        CallIdHeader callId = (CallIdHeader) response.getHeader(CallIdHeader.NAME);

        // ✅ 获取原始请求
        Request originalRequest = pendingRequests.get(callId.getCallId());
        if (originalRequest == null) {
            log.error("❌ 找不到原始MESSAGE请求，无法重发");
            return;
        }

        log.info("🔐 正在生成认证头...");

        // ✅ 获取 WWW-Authenticate 或 Proxy-Authenticate 头
        WWWAuthenticateHeader wwwAuthHeader =
            (WWWAuthenticateHeader) response.getHeader(WWWAuthenticateHeader.NAME);
        ProxyAuthenticateHeader proxyAuthHeader =
            (ProxyAuthenticateHeader) response.getHeader(ProxyAuthenticateHeader.NAME);

        // ✅ 生成认证头（MD5摘要算法）
        AuthorizationHeader authHeader;
        if (wwwAuthHeader != null) {
            authHeader = authHandler.generateAuthHeader(
                originalRequest, wwwAuthHeader, username, sipPassword,
                AuthorizationHeader.NAME);
        } else if (proxyAuthHeader != null) {
            authHeader = authHandler.generateAuthHeader(
                originalRequest, proxyAuthHeader, username, sipPassword,
                ProxyAuthorizationHeader.NAME);
        } else {
            log.error("❌ 没有找到认证头");
            return;
        }

        // ✅ 创建新的请求（复制原始请求）
        Request newRequest = (Request) originalRequest.clone();

        // ✅ 移除旧的Via头，创建新的Via头（新的branch）
        newRequest.removeHeader(ViaHeader.NAME);
        ViaHeader newViaHeader = getHeaderFactory()
            .createViaHeader(getLocalIp(), getLocalPort(), getTransport(), null);
        newRequest.addHeader(newViaHeader);

        // ✅ 添加认证头
        newRequest.addHeader(authHeader);

        // ✅ 更新CSeq序列号
        CSeqHeader cseq = (CSeqHeader) newRequest.getHeader(CSeqHeader.NAME);
        cseq.setSeqNumber(cseq.getSeqNumber() + 1);

        log.info("🔐 使用认证信息重发MESSAGE请求...");

        // ✅ 发送带认证的请求
        ClientTransaction newTransaction = getSipProvider().getNewClientTransaction(newRequest);
        newTransaction.sendRequest();

        log.info("✅ 已重发带认证的MESSAGE请求");

    } catch (Exception e) {
        log.error("❌ 处理认证失败", e);
    }
}
```

**3. 集成到响应处理流程**

在 `processResponse` 方法中调用认证处理器（第564-571行）：
```java
@Override
public void processResponse(ResponseEvent responseEvent) {
    Response response = responseEvent.getResponse();
    int statusCode = response.getStatusCode();

    if (Request.MESSAGE.equals(method)) {
        if (statusCode == Response.OK) {
            log.info("✅ [消息送达确认] 200 OK");
            CallIdHeader callId = (CallIdHeader) response.getHeader(CallIdHeader.NAME);
            pendingRequests.remove(callId.getCallId());  // 清除待重发请求
        } else if (statusCode == Response.UNAUTHORIZED ||
                   statusCode == Response.PROXY_AUTHENTICATION_REQUIRED) {
            log.warn("⚠️ [收到认证质询] {} {}", statusCode, response.getReasonPhrase());
            // ✅ 自动处理401/407认证
            handleAuthenticationChallenge(responseEvent);
        } else {
            log.error("❌ [消息发送失败] {} {}", statusCode, response.getReasonPhrase());
        }
    }
}
```

**4. SIP摘要认证算法实现**

在 `SipAuthHandler.java` 中实现标准的RFC 2617摘要认证：
```java
public AuthorizationHeader generateAuthHeader(
    Request request,
    WWWAuthenticateHeader challenge,
    String username,
    String password,
    String headerName
) {
    String realm = challenge.getRealm();
    String nonce = challenge.getNonce();
    String algorithm = challenge.getAlgorithm();  // 通常是MD5

    // 计算 HA1 = MD5(username:realm:password)
    String ha1 = md5(username + ":" + realm + ":" + password);

    // 计算 HA2 = MD5(method:uri)
    String method = request.getMethod();
    String uri = request.getRequestURI().toString();
    String ha2 = md5(method + ":" + uri);

    // 计算 response = MD5(HA1:nonce:HA2)
    String response = md5(ha1 + ":" + nonce + ":" + ha2);

    // 构造 Authorization header
    AuthorizationHeader authHeader = headerFactory.createAuthorizationHeader(scheme);
    authHeader.setUsername(username);
    authHeader.setRealm(realm);
    authHeader.setNonce(nonce);
    authHeader.setURI(request.getRequestURI());
    authHeader.setResponse(response);
    authHeader.setAlgorithm(algorithm);

    return authHeader;
}
```

## 问题3：视频加载速度差异 & 画面撕裂

### 问题描述
在视频通话测试中发现严重的性能不一致问题：
- **User100**: 视频画面需要**20秒**才能显示，用户等待期间只能看到黑屏
- **User101**: 视频画面**瞬间**就能显示（< 1秒），体验流畅
- **早期版本**: 视频画面出现严重**撕裂**（tearing）现象，像"老式电视信号干扰"
- **环境一致**：两个用户使用相同网络、相同硬件配置，问题令人困惑

### 根本原因分析

**问题1：摄像头设备ID硬编码导致设备错配**

在 `VideoCapture.java` 中，摄像头设备ID被硬编码为0（第45-46行）：
```java
// VideoCapture.java (早期版本 - 第45行)

}

public void initialize(int deviceId) throws FrameGrabber.Exception {
    log.info("初始化视频采集器: 设备 {}", deviceId);

    // 使用OpenCV采集

    grabber.setImageWidth(VIDEO_WIDTH);   // 640
    grabber.setImageHeight(VIDEO_HEIGHT); // 480
    grabber.setFrameRate(FRAME_RATE);     // 10 FPS

    grabber.start();  // ⏱️ 如果设备ID错误，这里会超时重试
}
```

**实际硬件环境差异**：
```
User100的设备列表：
- 设备ID=0: OBS Virtual Camera（虚拟摄像头）
- 设备ID=1: USB外置摄像头（真实摄像头）⭐ 应该使用这个

User101的设备列表：
- 设备ID=0: 内置摄像头（真实摄像头）⭐ 正确
```

**User100的加载过程分析**：
```
00:00 - 尝试打开设备ID=0（OBS Virtual Camera）
00:05 - 设备响应超时，第1次重试
00:10 - 设备响应超时，第2次重试
00:15 - 设备响应超时，第3次重试
00:20 - 最终成功建立连接（虚拟摄像头响应极慢）
       ❌ 用户已经等待了20秒！
```

**问题2：视频渲染缺少线程同步机制**

在 `VideoRenderer.java` 中，渲染实现使用简单的逐像素写入（第65-97行）：
```java
// VideoRenderer.java (当前实现 - 第65-97行)
public void renderFrame(Frame frame) {
    if (frame == null || frame.image == null) {
        return;
    }

    try {
        // 1. 从 Frame 提取 BGR 数据
        ByteBuffer buffer = (ByteBuffer) frame.image[0];
        byte[] data = new byte[buffer.remaining()];
        buffer.get(data);

        // 2. 转换 BGR -> RGB
        byte[] rgbData = convertBGRtoRGB(data, videoWidth, videoHeight);

        // 3. 更新 WritableImage（❌ 单缓冲区）
        PixelWriter pixelWriter = writableImage.getPixelWriter();
        pixelWriter.setPixels(
            0, 0,
            videoWidth, videoHeight,
            PixelFormat.getByteRgbInstance(),
            rgbData, 0,
            videoWidth * 3
        );

        // 4. 在 JavaFX 线程中绘制（⚠️ 有并发风险）
        Platform.runLater(() -> {
            graphicsContext.drawImage(writableImage, 0, 0,
                                     canvas.getWidth(), canvas.getHeight());
        });

    } catch (Exception e) {
        log.error("渲染视频帧失败", e);
    }
}
```

**并发问题时序分析**：
```
线程1（RTP接收）                线程2（JavaFX渲染）
│
├─ 接收到帧N
├─ 写入writableImage (行0-100)
│                                ├─ drawImage开始
│                                ├─ 读取行0-100 ✅ 正常
│                                ├─ 读取行101-200 ❌ 帧N+1数据！
├─ 接收到帧N+1                   │
```

**2. 优化渲染性能，确保线程安全**

改进后的渲染策略（虽然当前代码已使用单缓冲+Platform.runLater，但可以进一步优化）：

**关键改进点**：
- 使用 `PixelWriter.setPixels()` 一次性写入整个图像（已实现）
- 确保所有UI操作在 `Platform.runLater()` 中执行（已实现）
- 添加帧率控制，避免过度渲染：

```java
// 优化的渲染方法（建议增强版）
private long lastRenderTime = 0;
private static final long MIN_RENDER_INTERVAL = 33; // 30 FPS

public void renderFrame(Frame frame) {
    // 控制渲染帧率，避免过度渲染
    long currentTime = System.currentTimeMillis();
    if (currentTime - lastRenderTime < MIN_RENDER_INTERVAL) {
        return; // 跳过此帧
    }
    lastRenderTime = currentTime;

    // ✅ 在后台线程完成数据转换
    byte[] rgbData = convertBGRtoRGB(data, videoWidth, videoHeight);

    // ✅ 在JavaFX线程中一次性更新UI
    Platform.runLater(() -> {
        PixelWriter pixelWriter = writableImage.getPixelWriter();
        pixelWriter.setPixels(0, 0, videoWidth, videoHeight,
            PixelFormat.getByteRgbInstance(), rgbData, 0, videoWidth * 3);

        graphicsContext.drawImage(writableImage, 0, 0,
            canvas.getWidth(), canvas.getHeight());
    });
}
```

**3. 增强摄像头资源管理**

```java
public void stopCapture() {
    if (!isCapturing) {
        return;
    }

    log.info("停止视频采集");
    isCapturing = false;
```

## 问题4：会议未创建却可加入

### 问题描述
会议功能出现严重的**逻辑安全漏洞**：
- 用户A尝试创建会议，**创建失败**，客户端提示"会议创建失败"
- 用户B输入一个**随机会议ID**，**竟然可以加入成功**

```java
// ConferenceController.java (修复版)
@PostMapping("/join")
public Result<ConferenceInfo> joinConference(@RequestBody JoinRequest request) {
    String conferenceId = request.getConferenceId();
    Long userId = request.getUserId();

    // ✅ Step 1: 检查会议是否存在
    Conference conference = conferenceService.getById(conferenceId);
    if (conference == null) {
        log.warn("❌ 会议不存在: conferenceId={}", conferenceId);
        return Result.error(404, "会议不存在，请检查会议号");
    }

    // ✅ Step 2: 检查会议状态
    if (!"active".equals(conference.getStatus())) {
        log.warn("❌ 会议未激活: conferenceId={}, status={}",
                 conferenceId, conference.getStatus());
        return Result.error(400, "会议未激活或已结束");
    }

    // ✅ Step 3: 检查参与人数限制
    int currentParticipants = conferenceService.getParticipantCount(conferenceId);
    if (currentParticipants >= conference.getMaxParticipants()) {
        log.warn("❌ 会议人数已满: conferenceId={}, max={}, current={}",
                 conferenceId, conference.getMaxParticipants(), currentParticipants);
        return Result.error(400, "会议人数已满（" + currentParticipants + "/" +
                            conference.getMaxParticipants() + "）");
    }

    // ✅ Step 4: 检查用户权限（可选）
    if (conference.getIsPrivate() && !conferenceService.hasPermission(userId, conferenceId)) {
        log.warn("❌ 无权限加入私密会议: userId={}, conferenceId={}",
                 userId, conferenceId);
        return Result.error(403, "您没有权限加入此会议");
    }

    // ✅ Step 5: 添加参与者记录
    ConferenceParticipant participant = new ConferenceParticipant();
    participant.setConferenceId(Long.parseLong(conferenceId));
    participant.setUserId(userId);
    participant.setJoinTime(new Date());
    conferenceService.addParticipant(participant);

    log.info("✅ 用户成功加入会议: userId={}, conferenceId={}", userId, conferenceId);

    return Result.success(conference);
}
```

```java
/**
 * ✅ 新增：验证会议状态（HTTP API调用）

private boolean verifyConferenceBeforeJoin(Long conferenceId) {
    try {
        String url = SipConfig.getHttpServerUrl() + "/api/conference/" + conferenceId + "/verify";
        logger.info("验证会议状态: {}", url);

        java.net.URL apiUrl = new java.net.URL(url);
        java.net.HttpURLConnection conn = (java.net.HttpURLConnection) apiUrl.openConnection();
        conn.setRequestMethod("GET");

        // 添加认证Token
        if (authToken != null && !authToken.isEmpty()) {
            conn.setRequestProperty("Authorization", "Bearer " + authToken);
        }

        conn.setConnectTimeout(5000);
        conn.setReadTimeout(5000);

        int responseCode = conn.getResponseCode();
        logger.info("会议验证响应码: {}", responseCode);

        if (responseCode == 200) {
            logger.info("✅ 会议验证成功");
            return true;
        } else if (responseCode == 404) {
            logger.warn("❌ 会议不存在");
            return false;
        } else if (responseCode == 400) {
            logger.warn("❌ 会议未激活或已结束");
            return false;
        } else {
            logger.warn("❌ 会议验证失败: HTTP {}", responseCode);
            return false;
        }

    } catch (Exception e) {
        logger.error("❌ 会议验证异常: {}", e.getMessage(), e);
        return false;
    }
}
```
